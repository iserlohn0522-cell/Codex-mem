# Antigravity Memory (`ag-mem`)

## Overview
Antigravity is a sub-worker that safely reads and writes Codex memory to sync progress. Codex is the master authority. All memory entries created by Antigravity must be explicitly tagged and adhere to strict schemas to prevent corruption.

## Core Invariant
`memory is durable decision layer, not chronological notebook, not execution authorization.`

## Required Reading
**CRITICAL**: You MUST read `references/codex-schema-compliance.md` before your first write to any Codex memory file. It contains the mandatory JSONL schemas and markdown structures.

## Internal Modes
- `remember`: Save current operating state, human decisions, or durable constraints.
- `search`: Recover prior project memory context.
- `report`: Generate summaries or tagged artifacts based on memory.

## Operating Rules
1. **In-place updates**: Update `project_memory.md` within existing sections. Do NOT append to the bottom of the file.
2. **Inline Tagging**: Every line inserted into shared markdown files MUST start with `[AG]`.
3. **Artifact Naming**: Generated files to advance state MUST append `_ag` before the extension (e.g., `report_ag.md`).
4. **JSONL Schemas**: Use `source: "antigravity"` and `tags: ["ag"]` when writing to `.codex-mem/observations.jsonl`.
5. **No Destruction**: Never delete, trim, or rewrite content generated by Humans or Codex.
6. **Non-Authorization**: Memory is NOT execution authorization. Entries do not authorize submission, execution, or escalation.

## Red Flags — STOP and Start Over
If you catch yourself thinking any of these things, STOP. You are rationalizing a violation:

| Rationalization | Reality |
| :--- | :--- |
| "I'll rewrite `project_memory.md` cleanly to summarize everything." | No. In-place updates only. Never delete Codex history. |
| "I'll skip the `[AG]` tag, the context makes it obvious." | No. Every inserted line MUST have `[AG]`. |
| "I'll create a new top-level section in `project_memory.md` for my findings." | No. Use the 9 fixed Codex sections only. |
| "I'll write my long logs to a new `.ag-mem` folder." | No. All observations go to `.codex-mem/observations.jsonl` with proper tags. |
| "It says 'execute standard tests next', so I will run them." | No. Memory is not operational approval. Do not execute based on memory. |
| "The JSONL format is too restrictive, I'll add a `content` field." | No. Use the exact Codex JSONL schema. |

## Search Discipline
To avoid redundant work, retrieve memory in this exact priority order:
1. `project_memory.md`
2. `project_stage_log.md`
3. `.codex-mem/observations.jsonl`
4. `.codex-mem/execution_pointers.json`

## Protected Human Decisions
Never auto-overwrite or silently reinterpret decisions in the `Protected Human Decisions` section. If a proposed action conflicts with one, stop and ask the user.
